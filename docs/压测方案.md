# TaskFlow AI 压测方案

> 版本: 1.0 | 日期: 2026-05-22

---

## 一、压测目标

| 层级 | 目标 | 关键指标 |
|------|------|---------|
| HTTP 接入层 | 验证并发请求处理能力 | QPS、P99 延迟、错误率 |
| Agent 执行层 | 验证 ReAct 循环在高负载下的稳定性 | 步数分布、超时率、线程池耗尽 |
| Redis 缓存层 | 验证缓存命中率对响应时间的影响 | 命中率、缓存穿透次数 |
| PostgreSQL | 验证连接池与写入性能 | 活跃连接数、写入延迟、慢查询 |
| 异步 + SSE | 验证并发 SSE 连接与推送延迟 | 并发连接数、首字节时间 |

### 场景 2：同步任务执行（真实 LLM 调用，注意成本）

**目的**：验证完整 ReAct 链路的稳定性与性能基线。

⚠️ **成本提醒**：每次真实执行会调用 DeepSeek API（plan + action × N + reflect × N + synthesis ≈ 5-10 次 chat），加上 Tavily/WeatherAPI。建议控制执行次数。

| 用例 | 任务描述 | 并发 | 总次数 | 预期 |
|------|---------|------|--------|------|
| 2.1 单任务基线 | "计算 100 + 200" | 1 | 5 | 完成，< 5s/次 |
| 2.2 搜索任务 | "搜索 Java 25 新特性" | 1 | 5 | 完成，< 40s/次 |
| 2.3 多工具协作 | "查询太原天气并推荐散步地点" | 1 | 3 | 完成，< 35s/次 |
| 2.4 低并发 | 以上混合任务 | 3 | 9 | 全部完成，无超时 |
| 2.5 中并发 | 简单计算任务 | 5 | 15 | 全部完成，≤ 1 超时 |

```bash
# 单任务 curl 循环
for i in $(seq 1 5); do
  curl -s -X POST http://localhost:8080/api/v1/agent/execute \
    -H "Content-Type: application/json" \
    -d '{"task": "计算 100 + 200", "timeout": 60, "maxSteps": 5}' \
    | jq '{taskId, status, executionTimeMs}'
done

# 并发（需要 parallel 或 xargs）
seq 1 5 | xargs -P5 -I{} curl -s -X POST http://localhost:8080/api/v1/agent/execute \
  -H "Content-Type: application/json" \
  -d '{"task": "计算 {} + {} 的结果", "timeout": 60, "maxSteps": 3}'
```

**关键观察**：
- 所有任务的步数分布（是否出现 > 3 步的异常情况）
- 单次工具调用耗时分布（weather ~500ms, search ~1200ms, calculator < 5ms）
- LLM chat 调用的平均耗时（通常 2-5s/次）

### 场景 3：Mock LLM 高并发（JUnit 集成测试）

**目的**：绕过 LLM 成本限制，验证 Agent 引擎在高并发下的线程安全与资源管理。

用 `StubLLMClient` 替代真实 LLM，`MockWebServer` 替代外部 API，在 JUnit 中启动 Spring 容器并发执行。

```java
@Test
void concurrentTaskExecution() throws Exception {
    int concurrency = 20;
    int tasksPerThread = 5;
    ExecutorService executor = Executors.newVirtualThreadPerTaskExecutor();
    var latch = new CountDownLatch(concurrency);
    var failures = new AtomicInteger();

    for (int i = 0; i < concurrency; i++) {
        executor.submit(() -> {
            try {
                for (int j = 0; j < tasksPerThread; j++) {
                    TaskResult result = agent.execute("计算 " + j + " + " + j);
                    if (result.status() != ExecutionStatus.COMPLETED) {
                        failures.incrementAndGet();
                    }
                }
            } finally {
                latch.countDown();
            }
        });
    }

    assertTrue(latch.await(120, TimeUnit.SECONDS));
    assertEquals(0, failures.get());
    executor.shutdown();
}
```

| 指标 | 预期值 |
|------|--------|
| 并发线程数 | 20 |
| 总任务数 | 100 |
| 全部完成 | 是 |
| 失败数 | 0 |
| 线程池耗尽 | 否 |

### 场景 4：幂等性压测

**目的**：验证高并发下幂等键的原子性。

| 用例 | 并发 | 请求 | 预期结果 |
|------|------|------|---------|
| 4.1 同 Key 并发 | 10 | 相同 X-Idempotency-Key | 1 个 202，9 个 409 |
| 4.2 不同 Key 并发 | 10 | 不同 X-Idempotency-Key | 全部 202 |
| 4.3 混合场景 | 20 | 10 同 Key + 10 不同 Key | 11 个 202，9 个 409 |

```bash
# 同 Key 并发测试
KEY="test-idempotent-$(date +%s)"
for i in $(seq 1 10); do
  curl -s -o /dev/null -w "%{http_code}\n" -X POST http://localhost:8080/api/v1/agent/execute \
    -H "Content-Type: application/json" \
    -H "X-Idempotency-Key: $KEY" \
    -d '{"task": "计算 1+1", "timeout": 30, "maxSteps": 3}' &
done | sort | uniq -c
# 预期: 1 个 202, 9 个 409
```

## 五、关注瓶颈

| 组件 | 可能的瓶颈 | 监控方式 |
|------|-----------|---------|
| DeepSeek API | 速率限制、延迟波动 | 应用日志中 LLM chat 耗时 |

### 发现的问题及修复

1. **幂等性竞态条件** — `IdempotencyFilter` 的 `isDuplicate()` → `store()` 不是原子操作。并发下同一 key 的多个请求可同时通过检查，导致幂等失效。**已修复**：改用 `ConcurrentHashMap.putIfAbsent()`（本地）和 `SET NX`（Redis）实现原子 check-and-set。
2. **Flyway baseline-on-migrate** — 首次引入 Flyway 到已有 schema 的数据库时必须设置 `baseline-on-migrate: true`，否则 Flyway 拒绝在非空数据库上运行。**已配置**。
